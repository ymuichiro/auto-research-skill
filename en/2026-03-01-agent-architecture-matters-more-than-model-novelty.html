What Stood Out

The main signals visible at that point

The architecture layers have become visible

Taken together, the 2025 launches make it impossible to describe the agent stack as just a model. Protocols and SDKs handle connection, while runtimes and control planes handle operation.

Bounded autonomy is solidifying as the practical path

The strongest use cases are still concentrated around bounded workflows. What has changed is that those bounded workflows can now span more tools, data sources, and workers.

The comparison target is shifting from models to architectures

Even when comparing vendors, the question is no longer only which model is strongest. The differentiator lies in how protocol, developer tooling, approvals, evals, and interoperability fit together.

Use Cases

Use cases that already looked practical at that time

A multi-system enterprise assistant

  • Unify search, data access, notifications, ticket updates, and approval requests in one architecture.
  • A practical design keeps autonomy bounded while gradually expanding the number of connected systems.

A loop that connects development and research

  • Link issue understanding, supporting research, patch ideas, test coverage, and explanatory drafting into one loop.
  • In many cases, improving workflow visibility and human review stabilizes outcomes more than swapping the model.

Operating Implications

What had to be decided first in design, evaluation, and operations

Observed implication

By March 1, the needed translation is no longer from capability to possibility but from capability to architecture.

  • Design protocol, runtime, and control plane as separate layers so each can evolve independently.
  • Start with bounded workflows and expand autonomy only where recovery paths already exist.
  • Compare vendors not just on model and price but on how easily evaluation and approval can be implemented.

Key Takeaway

The provisional conclusion that could already be drawn

By March 1, 2026, the main question in agent adoption had moved a long way from which model to pick toward which architecture the organization can actually sustain.
