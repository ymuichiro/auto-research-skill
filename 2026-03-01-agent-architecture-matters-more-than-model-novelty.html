What Stood Out

この時点で見えていた主要シグナル

architecture の層が見えるようになる

2025年の各発表をつなぐと、agent stack は model ひとつで語れなくなっている。protocol と SDK が接続を担い、runtime と control plane が運用を担うという分担が見えやすい。

bounded autonomy が現実路線として定着する

最も筋のよいユースケースは、いまだに bounded workflow に集中している。違うのは、その bounded workflow がより多くの tool、data source、worker をまたげるようになったことだ。

比較すべきものは model より architecture になる

どのベンダーが強いかを考えるときも、問うべきは model だけではない。protocol、developer tooling、approval、evals、interoperability をどう束ねるかが差を作る。

Use Cases

その時点で現実味が高かったユースケース

multi-system enterprise assistant

  • 検索、データ参照、通知、チケット更新、承認依頼をひとつの architecture で束ねる。
  • bounded autonomy を保ちながら、対象システムだけ広げる設計が現実的である。

開発と調査をつなぐ loop

  • issue の理解、関連調査、修正案、テスト観点、説明文の下書きを連携させる。
  • model を替えるより、workflow の観測性と human review を厚くしたほうが成果が安定しやすい。

Operating Implications

設計・評価・運用で先に決めるべきこと

観測点

3月1日時点で必要なのは、agent を『何ができるか』で語ることではなく、『どの architecture で運用するか』に翻訳することだった。

  • protocol、runtime、control plane を別レイヤーとして設計し、入れ替えやすくする。
  • bounded workflow から始め、recovery path がある範囲だけ autonomy を広げる。
  • vendor comparison は model と価格だけでなく、evaluation と approval の実装容易性で見る。

Key Takeaway

この時点で引けた暫定結論

2026年3月1日の時点で、agent 導入の主論点は『どの model を選ぶか』より『どの architecture を維持できるか』へかなり移っていた。
