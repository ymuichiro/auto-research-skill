What Stood Out

この時点で見えていた主要シグナル

AgentKit と Agent Framework が共通部品化を進める

OpenAI と Microsoft の発表は、agent 開発を prompt craft ではなく workflow composition として扱う方向を強めている。graph、handoff、tooling を共通化する発想が前に出てきた。

長い state を持つ workflow をどう保つかが中心になる

単発実行では済まない flow が増えるほど、途中状態、再開、handoff 記録が重要になる。runtime の良し悪しは model choice では置き換えられない。

human checkpoint の設計が標準化し始める

workflow tooling が整ってくると、どの段階で人間が見るかをフロー上で表現しやすくなる。agent autonomy は、どこまで止められるかとセットで設計される。

Use Cases

その時点で現実味が高かったユースケース

オンボーディング・手続き orchestration

  • 必要情報の回収、文書生成、関係部署への連絡を順に下書きする。
  • 長い state を持つ flow なので、resume と handoff の設計が重要になる。

インシデント司令塔の補助

  • 状況整理、関係者通知、対応記録の下書きを agent が補助する。
  • human checkpoint を明示すれば、誤判断を避けつつ速度を上げやすい。

Operating Implications

設計・評価・運用で先に決めるべきこと

観測点

11月時点では、agent の複雑さに対抗するには、個別プロンプト最適化ではなく workflow tooling を持つことが重要になっていた。

  • 長い state を持つ flow では、resume と replay を標準機能として考える。
  • human review を業務外の運用でなく、flow 定義の一部に組み込む。
  • より単純な deterministic workflow へ落とせる区間は、無理に agent 化しない。

Key Takeaway

この時点で引けた暫定結論

2025年11月の時点で、agent 導入を前に進める鍵は model novelty ではなく、複雑さを扱える workflow tooling を持てるかになっていた。
