What Stood Out

この時点で見えていた主要シグナル

Google は protocol と platform を同時に押し出す

Vertex AI の multi-system agents と A2A は、単一ベンダー内に閉じない agent coordination を前提にしている。hosted platform を出しつつ、接続規約も外へ開こうとしている点が特徴的だ。

Anthropic と Microsoft は開発者 runtime を厚くする

Claude Code と Semantic Kernel Agents GA は、agent が単なる prompt template ではなく、開発ループやワーカー構成の一部として扱われる流れを強めた。SDK と host の関係が重要になる。

open と managed を組み合わせる設計が現実路線になる

この時点では、すべてを self-host するか、すべてを managed に寄せるかの二択ではなくなっている。protocol や SDK は open に寄せ、運用基盤は managed feature を使う、という混成構成が見え始めた。

Use Cases

その時点で現実味が高かったユースケース

コーディング支援の multi-tool ループ

  • コード探索、要約、テスト実行、レビュー観点整理を複数ツールで分担する。
  • 役割分離した worker を使う場合も、最終的な変更適用は人間が保持する構成が向く。

複数システムをまたぐ社内アシスタント

  • 検索、チケット、通知、承認系をつなぎ、回答案と次アクション候補を出す。
  • A2A や SDK の抽象があると、将来の接続先変更に備えやすい。

Operating Implications

設計・評価・運用で先に決めるべきこと

観測点

5月時点では、ベンダーごとの feature 差より、どこを open にし、どこを managed にするかの設計が中核になっていた。

  • 接続規約、tool schema、session 管理のどこを共通化するかを決める。
  • single agent で足りる工程と、role-based workers が必要な工程を分ける。
  • 将来のベンダー変更や接続先増加を見越し、強く結合しすぎない。

Key Takeaway

この時点で引けた暫定結論

2025年5月の時点では、agent stack は『単一製品の選定』ではなく、『接続規約と運用基盤をどう組み合わせるか』という設計問題になっていた。
