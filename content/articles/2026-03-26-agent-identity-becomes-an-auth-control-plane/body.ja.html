Signal Snapshot

agent identity は認証方式の部品ではなく control plane として扱われ始めた

2025年後半から 2026年3月までに公開された Microsoft Entra Agent ID、Amazon Bedrock AgentCore Identity、Google Cloud Agent Identity、 MCP、A2A、Auth0、Okta の一次情報を横断すると、AI agent は単なる service account の別名ではなくなりつつある。 公開資料から読み取れるのは、agent を first-class principal として発行し、必要に応じて user delegated authorization を重ね、 さらに protocol 上の auth metadata と governance を束ねる設計が、新しい比較軸として立ち上がっているという点である。 各社の成熟度は同じではないが、認証認可を agent runtime の外側の後付け設定にせず、control plane に寄せる方向はかなり明確だ。

4方向

収束している設計

native identity、delegated authority、protocol auth、governance overlay に整理できる。

3形態

principal の持ち方

agent 単体、user 委任、agent + user 併記の3パターンが主要である。

公開根拠

1結論

identity が実装責務を変える

今後は「どの token を使うか」より「誰の権限として動くか」を先に設計する必要がある。

Why This Matters

service account だけでは、agent の責任分界と委任の実態を扱いきれない

従来の app identity や service account でも API 呼び出し自体はできる。しかし agent の運用では、それだけでは粗すぎる。 なぜなら、agent は background task として自律的に動く一方で、ある場面では user consent に基づいて外部 SaaS や社内リソースへ触れ、 さらに MCP や A2A のように別の system や別の agent へ接続しにいくからだ。ここで必要になるのは、「誰の権限で動いたのか」を resource server と監査系の両方から追えること、そして権限の発行・交換・失効を prompt 内の約束事ではなく identity layer で扱えることだ。

共有 service account は権限が粗くなりやすい

複数 agent が同じ credential を使うと、least privilege と revoke 単位が崩れやすい。agent ごとに principal を分ける動きは、この問題への反応と読める。

user delegation だけでは agent 側の責任主体が消える

user token だけを見せる構成では、どの agent が代行したのかが resource server や監査ログから見えにくい。AWS や Google の docs は、agent と user を併記できる形へ寄っている。

protocol 接続では auth requirement の discoverability が必要になる

MCP や A2A のように動的接続が増えるほど、接続先がどの認証方式を要求するのかを metadata で伝える必要がある。これは個別 integration の外側にある設計責務である。

Four Directions

公開情報を並べると、agent identity は4方向へ整理できる

1. Native per-agent identity

  • agent 自体に固有 principal を発行し、owner、lifecycle、audit を紐づける方向である。
  • Microsoft Entra Agent ID、AWS AgentCore Identity、Google Agent Identity は、この層を explicit に扱っている。

2. Delegated user authority and tool access

  • agent が user consent や managed connection を通じて user 権限を借りる方向である。
  • Copilot Studio の action auth、AWS の 3LO と workload access token、Google の delegated OAuth、Auth0 の user-on-behalf パターンがここに入る。

3. Protocol-layer authorization

  • MCP と A2A は、接続先が必要とする auth requirement を protocol 上で discoverable にする。
  • MCP core auth、client credentials、enterprise-managed authorization、A2A の Agent Card / security scheme が代表例である。

4. Governance and lifecycle overlay

  • agent の所有者、公開範囲、認証方式、access review、shadow agent 検出を管理する層である。
  • Microsoft Agent Registry、Google IAM と logging、Okta の discovery / certification は、この方向の整備と見られる。

Vendor Map

各サービスを vendor 名ではなく「誰の権限で動くか」で並べる

Native per-agent identity

Principal: agent 単体

Grant method: agent 作成時または配備時に dedicated principal を発行

Policy enforcement point: Entra policy、IAM、AgentCore directory、resource 側 role

Representative services: Microsoft Entra Agent ID、Microsoft Agent 365、Amazon Bedrock AgentCore Identity、Google Cloud Agent Identity

Delegated user authority

Principal: user、または agent + user

Grant method: OAuth consent、3LO、managed connection、token exchange

Policy enforcement point: scope、connection policy、managed credentials、resource server 側 consent check

Representative services: Copilot Studio action auth、AWS supported auth patterns、Google delegated access、Auth0 AI agent flows

Protocol-layer authorization

Principal: agent client、enterprise auth broker、または service identity

Grant method: OAuth metadata discovery、client credentials、enterprise-managed authorization

Policy enforcement point: authorization server と downstream resource server

Representative services: MCP Authorization、MCP OAuth Client Credentials、MCP Enterprise-Managed Authorization、A2A specification、Auth0 for MCP

Governance overlay

Principal: owner / sponsor がひもづく登録済み agent

Grant method: registry 登録、access review、secret / connection 管理、shadow agent 検出

Policy enforcement point: registry、directory、audit log、certification workflow

Representative services: Microsoft Entra Agent Registry、Google IAM と logs、Okta の AI agent discovery / certification

How Permissions Are Granted

対向サービスは token の種類だけでなく principal の文脈を見て権限を与える

重要なのは、agent が token を持っていること自体ではない。対向サービスは、その token が誰に発行され、どの consent や policy を経てきたのかを見てアクセス可否を決める。 ここでは principal、credential source、resource server が検証するもの、権限制御の置き場所の4点で整理できる。

1. Agent principal only

  • principal: agent 単体
  • token / credential: agent 用に発行された JWT、access token、workload identity
  • resource server: issuer、audience、role、IAM policy を検証する
  • 権限制御: role、policy、resource binding で行う

バックグラウンド処理や narrow automation に向くが、user 文脈を必要とする操作にはそのままでは足りない。

2. User-delegated principal

  • principal: user
  • token / credential: OAuth consent で得た user-scoped token、managed connection
  • resource server: user の scope、consent、app registration を検証する
  • 権限制御: scope、delegated permission、connection policy に置く

support や finance 系の SaaS 連携で自然だが、どの agent が代行したかを別途残さないと監査粒度が粗くなる。

3. Agent + user combined context

  • principal: agent と user の両方
  • token / credential: token exchange、workload access token、delegated flow の監査ログ
  • resource server: user 権限を見つつ、どの workload / agent から来たかも検証する
  • 権限制御: scope と policy を二層で持つ

委任を保ちながら accountability を失わないための形であり、各社 docs がこの方向へ寄っているのは実運用上かなり重要である。

4. Central broker and registry

  • principal: enterprise control plane に登録された agent client
  • token / credential: enterprise-managed auth server や managed connection が発行
  • resource server: broker が出した token と advertised metadata を検証する
  • 権限制御: registry、authorization server、certification workflow に置く

MCP の enterprise-managed authorization や identity vendor 側の統制機能は、credential sprawl を減らしつつ revocation と監査をまとめる方向と読める。

Concrete Scenarios and Benefits

identity 設計が効くのは、高権限 workflow を narrow に切り出す場面である

M365

Microsoft 365 / Copilot 系 agent の社内アクセス

Microsoft 365 リソースや Copilot action を使う agent では、agent identity を固有 principal として持ちつつ、 必要な action だけ user auth を重ねる構成が現実的である。利点は、background task と user-mediated task を同じ agent で分けて管理できることだ。

SaaS

support / finance 系 SaaS への user 委任アクセス

CRM、ticketing、経費、請求のような system では、agent が user consent に基づいて限定スコープで API を呼ぶ形が向いている。 この方式の利点は、既存の権限モデルを保ったまま agent を導入しやすい点と、過剰権限の shared secret を避けやすい点である。

MCP

MCP / A2A で別 system や別 agent へ接続する

動的接続では、接続先がどの auth scheme を要求するかを metadata で配れることが重要になる。 利点は、client ごとの個別実装を減らし、enterprise 側で auth server や client credential policy を差し替えやすくなることだ。

Gov

shadow agent と credential sprawl を抑える

owner 不明の agent、散在する OAuth grant、長寿命 secret は、agent 数が増えるほど管理不能になりやすい。 discovery、registry、access certification を先に置く利点は、agent 導入を増やしても revoke と review の単位を保ちやすいことである。

この方式を取るメリット

principal を agent / user / broker に分けて管理すると、least privilege、auditability、revocation、delegation の4点を同時に改善しやすい。 これは security のためだけでなく、運用チームが agent を増やせる速度そのものに効く。

Operating Implications

導入時に先に決めるべき論点は、model より principal 設計に近い

観測点

公開資料がそろって示しているのは、agent auth の本質が「認証方式の比較」ではなく「権限主体と管理責務の切り分け」にあるということだ。

  • agent principal、user principal、connection owner を同一視しない。
  • 長寿命 secret を配るより、short-lived token、token exchange、managed connection を優先する。
  • 権限チェックは prompt や wrapper に埋めず、IdP、policy、resource server のどこで強制するかを明示する。
  • MCP / A2A の接続は、auth metadata と許可済み client の管理まで含めて設計する。
  • registry、discovery、certification を rollout の最後ではなく最初の control plane として扱う。

Key Takeaway

結論

agent 認証認可の比較軸は、単に OAuth を使うかどうかではなくなっている。公開資料から読み取れる現在地は、 agent を first-class principal として持ち、必要に応じて user delegation と protocol auth を重ね、その上に registry と certification を載せる方向である。 どの vendor が勝つかはまだ早いが、どの platform が agent を governable な identity として扱えるかは、すでに明確な選定軸になり始めている。
