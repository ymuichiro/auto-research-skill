What Stood Out

この時点で見えていた主要シグナル

評価単位が『うまい回答』から『完了できた仕事』へ移る

ReAct と Toolformer が基本形を示した後、WebArena、SWE-bench、OSWorld、BrowserGym は、評価の中心を実在タスクの完遂率へ押し出した。年末時点では、エージェントはチャット拡張ではなく作業システムとして見られ始めている。

公式発表もツール接続と監督を前提にし始める

AWS の Bedrock Agents、Google の Vertex AI Agent Builder、Anthropic の tool use と effective agents は、モデル単体よりもツール呼び出し、制御、承認、実行境界を前面に出している。実装論点が API 単体から構成設計へ移っている。

早期導入は境界の明確な業務に集約する

2024年末までの一次情報だけを見ても、成功しやすい用途は、使うツールが決まっていて、完了条件を書けて、失敗時に人間へ差し戻せる仕事である。『何でもできる AI』より、手順化できる作業への適用が現実的だった。

Use Cases

その時点で現実味が高かったユースケース

障害切り分けと修正案作成

  • issue、ログ、過去の変更履歴を読み、関連箇所と調査観点を先に絞り込む。
  • 修正候補やテスト観点の下書きまでは任せ、実変更やマージは人間承認に残す。

社内ナレッジ検索と定型アクション

  • 社内文書や FAQ を参照し、回答案や依頼文の下書きを作る。
  • 送信、更新、申請などの書き込み系操作は承認点を置き、権限の境界を保つ。

Operating Implications

設計・評価・運用で先に決めるべきこと

観測点

この段階では、モデル選定よりも『どの業務区間なら安全に測れるか』を決めるほうが、PoC の成否に直結していた。

  • モデル比較の前に、使わせるツールと禁止する操作を文章で決める。
  • 社内の代表タスクを 5〜10 件選び、完遂率とレビュー負荷で測る。
  • 書き込みや外部送信を伴う操作は、必ず人間承認点を先に置く。

Key Takeaway

この時点で引けた暫定結論

2025年の出発点で重要だったのは、賢そうに見えるデモを増やすことではなく、完了条件と責任境界を持つ小さな業務へ落とし込めるかだった。
