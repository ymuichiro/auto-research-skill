What Stood Out

この時点で見えていた主要シグナル

ブラウザ・computer use が製品言語に入る

Anthropic の computer use と OpenAI の Operator は、実環境操作を研究デモの外へ押し出した。まだ限定的ではあるが、ポータル操作や画面遷移を伴う仕事が正式な対象として扱われ始めている。

オーケストレーションは notebook から runtime へ移る

AutoGen v0.4 は、複数エージェントの会話を実験的なプロンプト連鎖ではなく、より durable な構成要素として整理している。ここで論点になるのは、誰が planner で、誰が executor で、どこで止めるかだ。

評価と監視が別レイヤーとして立ち上がる

Google の Vertex AI 側の発信を見ると、モデルやエージェントの評価が、アプリ完成後の付け足しではなく設計項目として浮上している。試すことと測ることを分ける必要が見え始めた。

Use Cases

その時点で現実味が高かったユースケース

ポータル横断の業務アシスタント

  • 複数の管理画面を巡回し、必要情報を集めて更新候補の下書きを作る。
  • 確定操作だけ人間が受け持つと、computer use 系の価値と安全性を両立しやすい。

コードベース調査アシスタント

  • 障害報告や issue を起点に、関連ファイル、テスト、過去の差分を探索する。
  • 長い調査手順を runtime 化し、途中結果や tool call を追跡できる構成が向く。

Operating Implications

設計・評価・運用で先に決めるべきこと

観測点

2月時点では、『何ができるか』以上に、『途中経過をどこまで観測できるか』が導入可否を分け始めていた。

  • read 権限と write 権限を分けて設計し、最初から同じ扱いにしない。
  • ブラウザや端末操作を評価する replay 可能なタスク群を持つ。
  • tool call の履歴を残し、失敗時の差し戻し手順を明文化する。

Key Takeaway

この時点で引けた暫定結論

ブラウザや端末に触れるエージェントは、単体モデルの強さだけで評価できない。durable orchestration と観測性が、ここから先の前提条件になる。
