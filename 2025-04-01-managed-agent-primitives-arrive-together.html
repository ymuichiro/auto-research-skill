What Stood Out

この時点で見えていた主要シグナル

OpenAI が agent building を API 群として束ね始める

new tools for building agents は、単一の completions 呼び出しではなく、tool use、responses、evals を含む構成として agent を扱う。これは、開発者体験が『会話生成』から『実行系』へ移ることを意味する。

AWS では multi-agent coordination が正式機能になる

Bedrock の multi-agent collaboration GA は、複数エージェントの協調を概念実証ではなく managed feature として扱い始めた。これにより、単一エージェントか複数エージェントかが設計オプションとして比較しやすくなった。

プラットフォーム選定は、そのまま設計選定になる

この時点では、memory、tool execution、approval、evaluation をどこまでネイティブに持つかで、ベンダーごとの差が見え始めた。『どのモデルが強いか』だけでは比較が足りない。

Use Cases

その時点で現実味が高かったユースケース

API 接続型の業務アシスタント

  • 社内検索、CRM、チケット、通知 API をまたいで情報を集め、次のアクション案を作る。
  • workflow を managed runtime に寄せると、承認点とログを統一しやすい。

調査と実行を分けた multi-agent ループ

  • 調査役、要約役、実行役を分けると、長いタスクの責務分界が明確になる。
  • ただし、役割を増やすほど評価軸と停止条件の設計が必要になる。

Operating Implications

設計・評価・運用で先に決めるべきこと

観測点

4月時点の論点は、モデル性能比較より、どの runtime と orchestration 抽象を採用するかに移り始めていた。

  • memory、tool execution、approval の責務がどこにあるかを比較項目に入れる。
  • single-agent で済む仕事と multi-agent が必要な仕事を分ける。
  • managed feature に依存する部分と、自前で制御したい部分を切り分ける。

Key Takeaway

この時点で引けた暫定結論

2025年4月の時点で、agent 導入は『モデルを呼ぶか』ではなく、『どの execution stack に乗せるか』を問う段階に入り始めていた。
