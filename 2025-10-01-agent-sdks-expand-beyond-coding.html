What Stood Out

この時点で見えていた主要シグナル

Claude Agent SDK が coding の外へ抽象を広げる

Anthropic の Agent SDK は、tool use や workflow 制御を、エディタ補助ではなく agentic app の基本部品として見せている。これにより、同じ抽象を業務アプリ側へ持ち込みやすくなる。

データ接続の抽象も application framework に近づく

Firestore support を含む MCP Toolbox の更新は、agent が document だけでなく transaction-aware なデータ基盤へ近づいていることを示す。接続方法が SDK と同じ議論に入ってくる。

open web stack の議論が control plane の必要性を強める

Agent Factory のメッセージが示すのは、agentic app を増やすほど、共通の session、identity、policy、handoff を支える control plane が必要になることだ。SDK 単体では足りない。

Use Cases

その時点で現実味が高かったユースケース

顧客対応のコーディネーション

  • 問い合わせ、顧客情報、ナレッジ、承認フローをつなぎ、返信案や次アクションを整える。
  • SDK と data connector を組み合わせると、チャットの外で動く業務フローへ広げやすい。

日程・移動・社内手続きの実務補助

  • 候補調査、必要情報収集、申請ドラフト作成までを自動化し、確定だけ人間が行う。
  • environment permissions を細かく分ければ、汎用 workflow にも展開しやすい。

Operating Implications

設計・評価・運用で先に決めるべきこと

観測点

10月時点では、SDK の選定は開発者体験だけでなく、共通 control plane をどう持つかの判断と結びついていた。

  • session、memory、policy を app 単位ではなく共通基盤で扱えるか確認する。
  • data connector と workflow runtime を別物として放置しない。
  • agentic app を増やす前に、identity と approval の共通ルールを作る。

Key Takeaway

この時点で引けた暫定結論

2025年10月の時点で、agent SDK は coding 支援の外へ広がり、汎用 workflow を動かす application layer の部品になり始めていた。
