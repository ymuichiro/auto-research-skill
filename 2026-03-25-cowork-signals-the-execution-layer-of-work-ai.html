Research Signal

Cowork は、職場AIが long-running agent system へ拡張し始めたというシグナル

Anthropic の Cowork と Microsoft の Copilot Cowork を起点に、Google Gemini Enterprise、Slack Agentforce、OpenAI の deep research / ChatGPT agent / Codex を並べると、職場AIは execution layer 単体へ移るというより、reasoning・execution・governance を束ねた long-running agent system へ拡張していることが見えてくる。公開資料と主要論文をもとに、各社のレイヤー構成、ユースケース、戦略差、実務上の制約まで整理する。

Signal Snapshot

Cowork は、職場AIが long-running agent system へ拡張し始めたことを示すシグナルである

公開資料を並べると、Cowork という名称を前面に出しているのは現時点で Anthropic と Microsoft が中心である。 ただし、実際に起きている変化は名称より広い。Google Gemini Enterprise、Slack の Agentforce、OpenAI の deep research、ChatGPT agent、Codex まで含めると、 市場は chat から単純に execution layer へ移るのではなく、reasoning、execution、governance を束ねた long-running agent system へ拡張し始めている。

2社

Cowork を明示ブランド化

Anthropic と Microsoft が Cowork を正面から打ち出している一方、他社は別名で近い製品面を展開している。

3層

Reasoning、execution、governance

今回の論点は execution 単体ではなく、どのレイヤーを主軸にして統合しているかにある。

5群

収束しつつ異なる製品群

Anthropic、Microsoft、Google、Slack / Salesforce、OpenAI は近い方向へ進みつつ、競争軸は明確に分かれている。

1変化

Single chat から long-running system へ

差が付くのは応答品質だけでなく、長時間タスク、ツール利用、監視、承認をどう束ねるかである。

Layer Map

実態は execution layer 単体ではなく、複数レイヤーの統合である

本稿では Cowork を一つの製品名として扱うが、実装面では複数レイヤーが重なっている。ここを分けて見ると、各社の製品責務と戦略差が読みやすくなる。

Reasoning layer

長時間の調査、推論、統合を担う層であり、OpenAI の deep research はここに最も近い。完遂性は高いが、業務システムを直接更新する execution とは区別して見るべきである。

Execution layer

実際のシステム操作、ファイル更新、ticket 起票、コード編集、テスト実行を担う層である。Copilot Cowork、Slack Agentforce、OpenAI の Codex / Codex app はこの層を強く体現している。

Hybrid layer

調査と実行の両方をまたぐ汎用層であり、ChatGPT agent は reasoning と action を一体で扱う代表例である。Anthropic Cowork や Google Agentspace の一部の面もここにかかる。

Governance layer

identity、permissions、audit、DLP、workspace control、safety boundary を担う層である。Microsoft Agent 365 と Frontier Suite はこの層を強く前面化し、Anthropic も逆に不足部分を明示している。

What Changed

方向性は収束しているが、各社の主軸レイヤーはかなり違う

共通しているのは long-running task、社内コンテキスト接続、アプリ操作、承認や監視を一つの製品体験に寄せようとしている点である。 ただし、何を核に据えているかは各社で大きく異なる。

Anthropic

  • Cowork は Claude Code の agentic architecture を Claude Desktop に広げた research preview であり、desktop environment と local execution を強く押し出している。
  • plan、subtasks、VM、parallel workstreams、file write-back を扱える一方、audit logs や Compliance API に乗らない点も公式に開示している。
  • つまり Anthropic は、agentic architecture の拡張を先に出し、その後に enterprise governance を詰めている。

Microsoft

  • Copilot Cowork は artifact-native execution を Word、Excel、PowerPoint、Outlook、Copilot Chat に埋め込み、Work IQ で context grounding を担保している。
  • 同時に Agent 365 と Frontier Suite を並べ、identity、registry、audit、DLP、security posture を control plane として一体化している。
  • Microsoft の主軸は、execution 能力そのものよりも、governance を含む enterprise integration の厚みである。

Google

  • Google Agentspace / Gemini Enterprise は search と knowledge access を起点に agent platform を組み上げ、assistant actions と connectors で action へ伸ばしている。
  • 主戦場は「どの agent が最も自律的か」より、「どれだけ enterprise knowledge に grounded されるか」にある。
  • そのため Google の競争軸は、execution layer 単体より knowledge platform としての厚みである。

Slack / Salesforce

  • Agentforce は agent を DM、channel mention、workflow trigger の中へ入れ、workflow-native な teammate として扱う。
  • Slack Actions と Slack Data により、conversation からそのまま更新や通知へつなぐ設計が明確である。
  • ここでの強みは general agent runtime ではなく、「flow of work」に agent を埋め込む UI と workflow の近さにある。

OpenAI

  • OpenAI は単一プロダクトで Cowork と同型の面を作るのではなく、複数レイヤーを別製品でカバーしている。
  • deep research は reasoning layer、ChatGPT agent は browser と apps を含む hybrid layer、Codex / Codex app はコード探索、編集、実行、テストを担う developer-specific execution layer に近い。
  • したがって OpenAI は、single cowork product ではなく、agent runtime と capability を多面展開して execution をカバーしていると見る方が正確である。

戦略差

  • 収束しているのは「long-running agent system を製品化する」という方向性であり、競争は単一の execution layer に集中しているわけではない。
  • 実際の差は、control plane を主軸にするか、knowledge base を主軸にするか、workflow embedding を主軸にするか、runtime capability を主軸にするかで分かれている。

Use Cases

ユースケースも「全部同じ」ではなく、どのレイヤーを使うかで分かれる

実務に落とすと、reasoning 系、execution 系、hybrid 系で向く仕事が変わる。ここを分けないと導入設計も誤りやすい。

1. 開発ワークフローの execution

  • OpenAI の Codex / Codex app は、repository exploration、code edit、command execution、test run、parallel agent work を一体で扱う。
  • これは knowledge work 全体ではなく、software development に特化した execution layer と見るのが適切である。

2. 会議準備と営業ブリーフ

  • Google Agentspace は企業データや外部情報を横断して briefing を作る系のユースケースが強く、Slack でも account brief や close plan が前面に出ている。
  • これは reasoning と retrieval が中心で、必要な範囲だけ execution をつなぐ設計が向く。

3. 調査メモ、表計算、資料作成

  • Copilot Cowork や ChatGPT agent は、調査だけで終わらず、Word、Excel、PowerPoint、files、spreadsheets を直接更新できる点が強い。
  • この領域は reasoning と execution の両方が必要なため、hybrid layer の価値が出やすい。

4. メール、予定、軽い業務調整

  • schedule meeting、draft email、calendar coordination のような仕事は、アプリ横断の軽い execution と progress visibility が重要になる。
  • ここでは「回答」よりも「依頼を最後まで処理する」ことが UX の中心になる。

5. IT / incident triage

  • Slack Agentforce は conversation context を使って incident を検知し、ticket を起こし、適切な相手へ通知し、report をまとめる系の workflow に向く。
  • 高リスク操作を人に残しつつ、初動の情報集約と handoff を automation する設計が現実的である。

6. 社内ナレッジ検索と action 接続

  • Google の connectors、Anthropic の plugins、OpenAI の apps は、knowledge retrieval を次の action と切り離さずに扱う。
  • この系統は検索単体ではなく、draft、update、handoff までを一つの依頼として束ねる設計に向く。

Why Now

背景にあるのは、agent research が「会話」より「完遂」と「統合」を重視してきたことだ

Cowork 的な製品面は突然出てきたわけではない。研究の焦点は一貫して、モデルが現実の環境でどこまで完遂できるかへ移ってきた。 そこへ enterprise connectors、permission controls、evals、audit requirements が乗り、ようやく product surface として見え始めた。

2022

ReAct

推論と行動を往復させる構成が広まり、LLM を回答器ではなく、外部環境に働きかける主体として扱う視点が広がった。

2023

Toolformer / WebArena

ツール利用と realistic web task が中心論点になり、prompt quality より tool-enabled completion の重要性が上がった。

2024

WebVoyager / SWE-bench / GAIA / OSWorld / tau-bench

ブラウザ、コード、PC 操作、user-agent-tool interaction まで評価対象が広がり、agent は複数レイヤーを束ねる system として測られ始めた。

2025

deep research / Operator / Agentforce / Agentspace

調査、browser action、workflow execution、agent gallery、connectors が製品面に現れ、reasoning と execution の統合が進み始めた。

2026

Cowork / Agent 365 / Codex app

現在見えているのは、execution layer 単体の競争というより、reasoning、execution、governance をどこまで一つの long-running agent system として組み上げられるかの競争である。

この根拠から読み取れること

AI の進化は chat から execution layer へ単純に移ったのではなく、複数レイヤーを統合する long-running agent system へ拡張していると捉える方が正確である。

Operating Implications

実務では「全部 execution」と見なさず、どのレイヤーを設計対象にするかを先に分ける必要がある

方向性は収束しているが、導入の論点はベンダーごとに同じではない。reasoning を主に使うのか、system execution を主に使うのか、governance をどの control plane に乗せるのかで設計が変わる。

1. レイヤーを混同しない

deep research を execution の代表例にすると、業務システム更新の設計論点がぼやける。推論、実行、統制は別々に評価すべきである。

2. 主軸レイヤーでベンダーを選ぶ

Microsoft は governance、Google は knowledge platform、Slack は workflow embedding、OpenAI は runtime capability、Anthropic は desktop expansion が強い。比較表は同一軸一枚では足りない。

3. Deep research は前段として使う

deep research は long-running task を完遂するが、業務システムを直接更新する execution ではない。実務では「execution につながる前段の reasoning layer」として扱う方が設計しやすい。

4. 開発 execution は別カテゴリで見る

Codex / Codex app は明確に execution だが、その対象は software development である。knowledge work 全般と一緒にすると、必要な権限や評価軸がずれる。

5. Governance は capability と同格で見る

Agent 365、Frontier Suite、workspace controls、Anthropic の safety 制約は、agent 導入が model selection だけでは完結しないことを示している。

6. Narrow workflow から始める

meeting prep、incident summary、code fix、sales brief のように成果物と review point が明確な業務から始めると、どのレイヤーが効いているかを切り分けやすい。

Practical Constraints

導入の難しさは capability だけでなく、運用制約にもある

Cost

長時間実行、ツール呼び出し、browser 操作、外部 API 利用が増えるほど、token と execution cost は膨らむ。推論時間の長さはそのまま ROI 設計の論点になる。

Reliability

multi-step 実行では、早い段階の誤りが後段へ伝播しやすい。task completion だけでなく、step-level monitoring と retry policy を別で設計する必要がある。

UX

long-running agent system では、進捗可視化、中断、再指示、branching、handoff が重要になる。会話 UI だけでは十分でない場面が増える。

Operations

monitoring、human-in-the-loop、approval、incident response、retention、compliance は capability とは別の実装課題である。ここを軽視すると本番導入が止まりやすい。

Key Takeaway

結論

Cowork は、単なる新しい execution layer の名前ではない。より正確には、職場AIが reasoning、execution、governance を統合した long-running agent system へ拡張し始めたことを示すシグナルである。 Cowork 名称そのものは限定的だが、背景にある方向性は広く共有されている。これからの比較軸は「誰が smartest か」だけではなく、「どのレイヤーを主軸にし、どう安全に統合するか」になる。
